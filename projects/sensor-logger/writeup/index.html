Technical Write-up

Arduino Sensor
Data Logger

A multi-sensor environmental monitoring system built on the Arduino Uno, logging temperature, humidity, and barometric pressure to SD card over I²C. This write-up covers the hardware design decisions, firmware architecture, ADC noise issues encountered, and what I'd do differently on the next revision.

Overview

This project started as a deliberate first step toward custom PCB design. Before committing to KiCad and a fabricated board, I wanted to force myself to deal with real hardware communication protocols, power supply constraints, and file system management on a microcontroller — the kind of friction that breadboard prototyping is good at surfacing.

The system reads temperature, humidity, and barometric pressure from a Bosch BME280 sensor over I²C, and writes timestamped CSV rows to a FAT-formatted SD card every five seconds. The design is intentionally minimal: no display, no wireless, no RTOS. Just sensors, storage, and a reliable main loop.

Hardware Design

Component Selection

The BME280 was chosen over the simpler DHT22 for two reasons: it communicates over I²C rather than a custom single-wire protocol, making it representative of the kind of sensor interface I'd encounter in real designs; and it adds barometric pressure, which the DHT22 lacks entirely.

Wiring & Pull-ups

The Arduino Uno's Wire library enables internal pull-ups on SDA and SCL automatically, which is sufficient for a short bus with a single device at standard mode (100 kHz). I added external 4.7 kΩ resistors initially but found they made no measurable difference at this bus speed and cable length. For future designs with multiple I²C devices or longer runs, external pull-ups sized to the bus capacitance will be necessary.

Power

The board runs on USB 5V with the BME280 and SD module on the Arduino's 3.3V regulated rail. Under load the rail measured approximately 3.08V — a 0.22V sag that turned out to be significant for the ADC (covered in the issues section).

Firmware Architecture

Main loop structure

The firmware follows a simple fixed-interval polling pattern. A non-blocking millis() timer fires the sensor read and SD write every 5000ms without blocking other work. The main loop stays free for any future additions like a serial debug output or a button input.

main loop — sensor_logger.inounsigned long lastLog = 0; const unsigned long INTERVAL = 5000; void loop() { if (millis() - lastLog >= INTERVAL) { lastLog = millis(); logReading(); } // room for other non-blocking work here } void logReading() { File f = SD.open("log.csv", FILE_WRITE); if (!f) { handleSDError(); return; } sprintf(buf, "%lu,%.2f,%.2f,%.2f\n", millis(), bme.readTemperature(), bme.readHumidity(), bme.readPressure() / 100.0F); f.print(buf); f.close(); }

SD card error handling

The initial version used while(!SD.begin()) which caused the sketch to hang indefinitely if the card was missing at boot. The final version retries up to five times with a 500ms delay, then enters a safe idle loop that blinks the built-in LED rather than blocking execution entirely.

Lesson learned Never use a blocking while(!peripheral.begin()) pattern in production firmware. Always pair init failures with a timeout, a fallback state, and a visual indicator.

Issues & Solutions

ADC noise on analog inputs

The most significant problem was noisy analog readings from the capacitive moisture sensor on A0. Raw ADC values varied by ±80 counts in a stable environment — far too much for useful data. Investigation revealed two contributing factors.

First, the 3.3V rail was sagging under load, which shifts the ADC's effective reference voltage. The ATmega328P's ADC uses VCC as its default reference, so any ripple on the supply rail directly contaminates readings. Second, the SD card module was causing SPI bus activity that coupled noise into the analog circuitry.

Fig. 2 — Raw A0 readings before and after adding the 100µF bulk capacitor across VCC/GND and connecting AREF to the 3.3V rail.

The fix had two parts: a 100µF electrolytic capacitor across the power rails to dampen supply sag, and connecting the AREF pin to the 3.3V rail with a 100Ω series resistor to give the ADC a stable, independent reference. This brought variation down to ±6 counts — acceptable for this application.

Results

The completed system runs reliably over multi-hour sessions. The SD card log produces clean CSV output that can be plotted directly with the included Python script. A 24-hour overnight run captured 17,280 samples with no dropped entries or filesystem errors.

What I'd Do Differently

The most important lesson was about power design. I treated the power supply as an afterthought and paid for it with hours of debugging ADC noise. On the next revision — the custom PCB — I'll add proper decoupling capacitors at every IC, a dedicated 3.3V LDO with good rejection, and a dedicated analog power plane separated from the digital circuitry.

I'd also move away from the Arduino SD library toward a more lightweight implementation. The library opens and closes the file on every write, which causes a noticeable 30–40ms stall each sample interval. A better approach is to keep the file handle open and call flush() periodically rather than close/reopen.

Finally, timestamping with millis() is adequate for relative timing but produces logs that are hard to correlate with real-world events. A DS3231 RTC module would give wall-clock timestamps and is the obvious next addition.
